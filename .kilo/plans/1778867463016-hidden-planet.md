## Conversion approach per DOCX template

### Pipeline (applied by `scripts/convert_docx.php`)

```
DOCX file
  │
  ▼  [1 – extract]
PhpWord IOFactory::load() + HTML writer → raw HTML
Strip: <!DOCTYPE>, <html>, <head>, <style> block, <meta>, <title> tags
Strip: xmlns, lang, xml:lang attributes from root
Strip: <br /> → ""
Strip: &nbsp; → " "
Collect: extracted CSS block → save as {slug}.css
  │
  ▼  [2 – classify]
Detect pattern from metadata flags:
  bgc=has_tables+colspan+rowspan vs table=0
Identify first font pair (nameFont, bodyFont) from first <span style>
  │
  ▼  [3 – segment]
From top-to-bottom, identify sections:
  1. Header block  — name + role + contact
  2. Profile/Summary block  — body paragraphs after header until next section header
  3. Experience block  — separated by section-header marker
  4. Education block  — same
  5. Skills/Languages block
  6. Certifications block
  7. References (optional)
  │
  ▼  [4 – entryise (normalise structured blocks)]
  Experience/Education:
    → Table-based rows: each <tr> → <div class="cv-experience-entry">
      Parse: date<title>, company, description fields by heuristics
    → Flat-paragraphs: consecutive <p> pairs split by blank-line → entries
  Skills:
    → Comma/pipe-list in <span>/<p> → split → <div class="cv-skill-item">
  Languages:
    → "English — Native" pattern → <div class="cv-language-entry">
  Certifications:
    → "Cert Name - Issuer, Year" pattern → <div class="cv-certification-entry">
  │
  ▼  [5 – template skeletonise]
Wrap in:
  <div class="cv-{slug}">
    <header class="cv-{slug}-header"> …name…role…contact… </header>
    {# each detected block #}
    <section class="cv-{slug}-section cv-{slug}-experience">
      <h2 class="cv-{slug}-section-heading">Experience</h2>
      {{experience_entries}}
    </section>
    … <section class="cv-{slug}-section cv-{slug}-education"> …education… </section>
    … <section class="cv-{slug}-section cv-{slug}-skills"> …skills… </section>
    … <section class="cv-{slug}-section cv-{slug}-languages"> …langs… </section>
    … <section class="cv-{slug}-section cv-{slug}-certifications"> …certs… </section>
    … <section class="cv-{slug}-section cv-{slug}-references"> …ref… (if any)
  </div>
  │
  ▼  [6 – CSS clean & namespace]
A. remove all external style links (@import, link[rel=stylesheet] href=)
B. extract colour palette: brandColour (from bgcolor on header cell), sectionBg1 (1st section table header), sectionBg2 (2nd section), bodyBg, bodyText
C. build replacements:
   oldColour = sectionBg1 → newColour = built from brandColour (lightened)
   oldColour = sectionBg2 → newColour = built from brandColour (different shade)
D. remove common boilerplate selectors:
   body →, td {border: 1px…}, table {border-spacing: 0}, a.NoteRef
E. remove @font-face with remote url; keep local('…') fallback only
F. scope all selectors under .cv-{slug}
   conversion: selector { token }  →  .cv-{slug} selector { token }
   convert each selector: "h1" → ".cv-{slug} h1"
   "table" → ".cv-{slug} table"
   "td" → ".cv-{slug} td"
   ".xyz" → ".cv-{slug} .xyz"
   "#XXX" → drop (id-based selectors, not in play here)
G. inline bgcolor on <th>/<td> already lifted into .cv-{slug}-section-heading rules
H. normalise inline style parity → remove all remaining style= attributes
── after pass, run sed toolkit against and verify no duplicate key signatures
  │
  ▼  [7 – output]
{slug}.html — template HTML skeleton with {{placeholders}}
{slug}.css — clean namespaced CSS
seed_{slug}.sql — INSERT statement record
```

## Colour-variant de-duplication rule

For each family with N colour-difference files, **one template record** is created:
- Admont: 3 variants → 1 template (`admont`)
- Quebec: 3 variants → 1 template (`quebec`), colour encoded in bgcolour props  
- Tilikum: 3 variants → 1 template (`tilikum`)
- Zhoushan: 3 variants → 1 template (`zhoushan`)
- Peckham: 3 variants → 1 template (`peckham`)

Colour variation is captured in the CSS via `--accent`, `--header-bg`, `--section-heading-bg`, `--body-text` CSS custom properties on `.cv-{slug}`. End-users wishing the exact colour variants they saw in the original can switch templates (distinct slugs) rather than relying on warm/cold variants of one.

## Execution order

1. **Classify all 46 docx** — run `scripts/convert_docx_lite.php --classify` → `classified_docx.json`; review with user
2. **Show distinct families** — present the unique structural families; user selects which families to proceed
3. **Write skeleton script** `scripts/convert_docx_worker.php` — test with single template (e.g. `Khaju`)
4. **Iterate on first template** — review generated `{slug}.html` and `{slug}.css`, validate in browser at `/staging/`
5. **Batch convert** — loop all selected families via worker
6. **Write gallery + create mini-previews** for the new docx templates
7. **Write `migration_docx_templates.sql`** — INSERT all new records
8. **Apply migration to `cv_builder_staging`** — verify via `SELECT slug FROM cv_templates`
9. **Spot-check preview + PDF export** with demo data for each converted template
10. **Deploy** — `/var/www/deploy.sh deploy staging && deploy production`

## Notes

- `resume.html` / `resume.css` are plain HTML+CSS (not DOCX); same pipeline applies, slug=`yui-resume`
- WYSIWYG editor ignores template HTML/CSS (pre-existing limitation), out of scope
- No per-skill level is stored in the app model so skill **bar** markup is dropped in favour of clean `<div class="cv-skill-item">` tags; if specific job requires real percentage bars, that belongs to a separate commit post templates
- Dompdf renders inline `<p>` reliably; `<span>`-only skill blocks must emit `<p>` or `<div>` wrapper
